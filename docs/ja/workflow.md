# ワークフロー：ステップバイステップガイド

## 概要

```
ストーリー → ALPS → Fake → 合意 → スキーマ → Be実装
```

各ステップには明確なインプット、アウトプット、そして答えるべき問いがある。

---

## ステップ1：ストーリー

**問い：** ユーザーは何をしたいのか、なぜか？

**形式：** 構造化された自然言語のユーザーストーリー

```markdown
ユーザーとして、
[アクション]したい。
なぜなら、[理由]だから。
```

**アウトプット：** すべてのユーザーストーリーをまとめた `story/main.md`。アプリの概要とコアエンティティも含める。

**ポイント：**
- システムの視点ではなく、ユーザーの視点で書く
- 「なぜなら」節は必須——ドメインの真の意図が現れる
- 最後にエンティティを列挙する：このドメインにはどんな概念が存在するか？

**例：** [`example/story/main.md`](../../example/story/main.md) を参照

---

## ステップ2：ALPS

**問い：** アプリは何ができるのか、どんな語彙を使うのか？

**ツール：** [ALPS](https://alps.io/)（Application-Level Profile Semantics）

**アウトプット：** `alps/[name].json` ファイル。以下を定義する：
- **セマンティクスディスクリプタ：** データ要素（id, title, memo, isCompleted...）
- **safe ディスクリプタ：** 読み取り操作（goTodoList, goTodoDetail...）
- **unsafe/idempotent ディスクリプタ：** 書き込み操作（doCreateTodo, doCompleteTodo...）
- **表現：** 複合ビュー（TodoList, TodoDetail...）

**ポイント：**
- 可能な限り `def` でSchema.orgなどの標準語彙にリンクする
- 命名規則：safe（読み取り）は `go` プレフィックス、unsafe/idempotent（書き込み）は `do` プレフィックス
- ALPSファイルのバリデーション：[https://alps-io.github.io/](https://alps-io.github.io/)
- 状態遷移図の可視化：[app-state-diagram](https://github.com/koriym/app-state-diagram)

**例：** [`example/alps/todo.json`](../../example/alps/todo.json) を参照

---

## ステップ3：フェイクデータ

**問い：** このドメインは現実にどう見えるか？

**目標：** 実際のユーザーが作るようなリアルなフェイクデータを50件以上生成する。

**アウトプット：** リアルなサンプルデータの `fake/data-50.json`（または `.jsonl`）

**生成方法：**
AIモデルを使って以下のようなプロンプトで生成する：
```
50件のリアルなフェイクTodoアイテムを生成してください。各アイテムには：
- todoId: ULID形式（26文字）
- todoTitle: 短め・中くらい・長めなど様々なリアルなタイトル
- todoMemo: 時々null、時々短いメモ
- isCompleted: true/falseを混在
- createdAt: 過去3ヶ月のISO 8601タイムスタンプ

ランダムなテストデータではなく、実際のユーザーが作るようなデータにしてください。
```

**観察すること：**
- テキストフィールドの長さの分布
- オプションフィールドのnull/空の頻度
- 値のパターンと範囲
- 自然に現れるエッジケース

**例：** [`example/fake/data-50.json`](../../example/fake/data-50.json) を参照

---

## ステップ4：合意

**問い：** このデータはドメインに対する全員の理解と一致しているか？

**活動：** ステークホルダー（プロダクトオーナー、デザイナー、他の開発者）とフェイクデータをレビューする。

**聞くべき質問：**
- これらはリアルなtodosに見えるか？
- 実際のユーザーが必要とするものが足りないものはあるか？
- 実際のユーザーが絶対にしないことがあるか？
- 現実的な制限はどこか？（最大長、必須 vs オプション...）

**アウトプット：** ドメイン制約についての共通理解。ノートとして、またはスキーマの決定として直接文書化する。

**このステップが重要な理由：**  
ここで発見された誤解は修正コストがゼロに等しい。実装後に発見された誤解はすべてのコストがかかる。

---

## ステップ5：スキーマ

**問い：** このドメインについて合意した制約は何か？

**形式：** JSONスキーマ（表現ごとに1ファイル）

**アウトプット：** `schema/` ディレクトリのファイル群、主要なデータ構造ごとに1ファイル。

**フェイクデータから制約を導く方法：**

| 観察 | スキーマの決定 |
|------|--------------|
| 観察された最長タイトル：41文字 | `maxLength: 80`（実用値＋バッファ） |
| 多くのtodosにメモがない | `memo` はオプション |
| todoIdは常に26文字 | `minLength: 26, maxLength: 26` |
| isCompletedは常にboolean | `type: boolean` |

**ポイント：**
- `maxLength` は観察された最大値 × 約2倍にする（255のような慣習的なデフォルトではなく）
- フェイクデータで常に存在することが示されたフィールドのみ `required` にする
- 一貫性のためにALPSディスクリプタIDをプロパティ名として使う

**例：** [`example/schema/`](../../example/schema/) を参照

---

## ステップ6：Be実装

**問い：** このドメインをBeフレームワークでどうコード化するか？

**インプット：** ステップ5のスキーマが `Being` の定義になる。

**構造：**
```
src/
├── Being/
│   ├── TodoList.php      ← schema/todo-list.json に対応
│   └── TodoDetail.php    ← schema/todo-detail.json に対応
├── Final/
│   ├── CreateTodo.php    ← doCreateTodo
│   ├── CompleteTodo.php  ← doCompleteTodo
│   └── DeleteTodo.php    ← doDeleteTodo
└── Reason/
    ├── TodoRepository.php
    └── Todo.php          ← エンティティ
```

**接続：**
- ALPS `descriptor` id → Being のプロパティ名
- ALPS `safe` ディスクリプタ → Being（読み取り）
- ALPS `unsafe/idempotent` ディスクリプタ → Final（書き込み）
- JSONスキーマの制約 → Being のバリデーションルール

ドメインを先に設計したからこそ、コードはドメインのように読める。

---

## 完全なループ

6ステップすべてを完了すると、手元に残るもの：

1. どのステークホルダーも読んで確認できる**ストーリー**
2. アプリケーションの語彙を文書化した**ALPSプロファイル**
3. ドメインを具体的・レビュー可能にした**フェイクデータ**
4. 全員が同じ理解を共有したことを示す**合意**
5. 観察した現実を制約として記録した**スキーマ**
6. ドメインを映した**実装**

各ステップのアウトプットが次のステップのインプットになる。ストーリーがALPSを形作る。ALPSがフェイクデータを形作る。フェイクデータがスキーマを教える。スキーマが実装を導く。

無駄なステップはない。スキップできるステップもない。
