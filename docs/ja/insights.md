# 考察：観察と振り返り

## ドメインはパターンの先にある

ソフトウェア設計によくあるアプローチがある：デザインパターンを学び、それをドメインに当てはめる。

結果として生まれるコードはパターンに形作られ、ドメインはパターン型の容器に押し込まれる。

より良い方向は逆だ：ドメインを深く理解し、パターンがその理解から浮かび上がるようにする。

これは優れたフレームワークに当てはまる。BEAR.Sundayは「MVCとリソース指向を実装しよう」からは生まれなかった。ウェブが実際にどう機能するか——リソース、ハイパーメディア、状態遷移——を観察し、その現実に構造を決めさせることで生まれた。

ALPSは「APIディスクリプション形式を作ろう」からは生まれなかった。「アプリケーションは実装と独立に何を*意味*するのか」という問いから生まれた。

**be-semanticはプロジェクトレベルで同じ原則を体現する。** ドメインを観察し——ユーザーストーリー、リアルなデータ、ステークホルダーの理解——スキーマがその観察から浮かび上がるようにする。

パターンはドメインに従う。逆ではない。

## 50件の規律

なぜ50件のフェイクデータなのか？5件や10件ではなぜダメなのか？

5件だと、意図的に作りすぎてしまう。「短いタイトルを1件、長いタイトルを1件、メモあり1件、メモなし1件」と考える。まだ規定している。

50件だと、リアルにならざるを得ない。一つひとつを丁寧に作ることはできない。「50人の実際のユーザーが実際に作るものは何か？」と問わなければならない。そして50件のリアルなデータを眺めると、自分が置いていないパターンが現れる。

最長のタイトルは41文字だ。「タイトルは最大41文字」と決めたからではなく、50件のリアルなtodosを想像すると、誰も100文字のタイトルを書かないから。

50件中35件でメモがnullだ。「メモはオプション」と決めたからではなく、ほとんどのtodosにはメモが不要だから。

これが制約を規定することと観察することの違いだ。50件の演習は観察モードに強制的に切り替える。

## 合意ステップは最も軽視されている

個人プロジェクトでは合意ステップをスキップしたくなる。自分でフェイクデータを生成して、自分でレビューして、何を決めたか知っている。

しかし合意ステップはデータについてではない。共通理解についてだ。

プロダクトオーナーが50件のフェイクtodosを見て「これで正しい」と言うとき、暗黙的に確認している：
- 語彙（「title」と「memo」が意味をなすこと）
- 構造（todoにはこれらのフィールドがあり、他にはないこと）
- エッジケース（memoがnullでも問題ないこと）

プロダクトオーナーが50件のフェイクtodosを見て「あれ、期限日はどこ？」と言ったとき——それはコードレビューで見つけるよりはるかに安価な発見だ。

**フェイクデータは安価なコミュニケーションツールだ。** 抽象を具体にする。ステークホルダーに、読めないスキーマではなく、反応できるものを与える。

## スキーマは規定ではなく記録

フェイク＋合意ステップなしに書かれたJSONスキーマは推測だ。

それらのステップを経て書かれたJSONスキーマは記録だ。

この違いは重要だ。「推測スキーマ」は、実際のユーザーが実際のデータを送ったときにのみ現れる微妙な誤りを含む。「記録スキーマ」は、チームが実際に合意したドメインの姿を捉える。

`maxLength: 255` は推測だ（またはデータベースのデフォルトからの習慣）。  
リアルなタイトルが最大41文字であることを観察した上で導いた `maxLength: 80` は記録だ。

これはエッジケースへの対応も変える。誰かが「なぜ maxLength は80なのか？」と聞いたとき、こう答えられる：「50件のリアルな例を見て最長が41文字だったので、余裕を持って2倍の80にしました」。これは根拠のある、追跡可能な決定だ。

## スキップのコスト

ワークフローの各ステップにはコストがある。スキップにもコストがある。

| スキップすると... | リスク... |
|----------------|-----------|
| ストーリー | 誰も必要としない機能を実装する |
| ALPS | 共通語彙がない；同じ言葉で異なることを意味する |
| フェイク | 現実に合わない制約；本番での予期せぬエッジケース |
| 合意 | コードレビューまたはリリース後に誤解を発見する |
| スキーマ | 実装のドリフト；開発者ごとに異なる前提 |
| Be実装 | 実装がドメインモデルから乖離する |

ワークフローは各ステップが次のステップを簡単・安価にするように設計されている。

## Be Frameworkとの関係

Be Frameworkの哲学は「Be, Don't Do（する、ではなく、ある）」だ。オブジェクトは何かを*する*のではなく、何かで*ある*べきだ。

これはbe-semanticと自然に合致する：

- `ストーリー` はユーザーが何で*ありたいか*を定義する（整理されたtodosを持つ人）
- `ALPS` はアプリケーションが何で*あり得るか*を定義する（様々な状態で）
- コードの `Being` は*あるもの*を表す（todo一覧、todo詳細）
- `Final` は存在の新しい状態への遷移を表す

ワークフロー全体が、*行動のシーケンス*ではなく、*存在の状態*を中心に向いている。

設計が意味——物事が*何であるか*——から始まるとき、Be実装はその意味の自然な表現になる。
