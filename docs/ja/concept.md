# コンセプト：意味から設計する

## 核心となる考え方

ソフトウェア開発には、出発点を間違える習慣がある。

フレームワークを選び、データベースを決め、テーブルを設計する——ドメインを深く理解する前に。その結果、コードは技術の言葉で書かれ、ドメインの言葉では書かれない。

**be-semantic** は別の出発点を提案する：*意味*から始めよ。

「どうやって保存するか？」「エンドポイントはどう見えるべきか？」を問う前に、こう問う：

> ユーザーは何をしたいのか？  
> その行為は現実世界でどういう*意味*を持つのか？  
> その意味には、実際にどんな制約があるのか？

## 構造の前にセマンティクス

「セマンティクス（semantics）」という言葉はギリシャ語の *sēmantikós*——「重要な、意味を持つ」——に由来する。

ソフトウェアにおけるセマンティクスとは、「何かがどう実装されているか」ではなく、「何かが何を意味するか」のことだ。

`title` というフィールドを例にとろう：
- データベース設計者は言う：`VARCHAR(255)` — それがデフォルトだから
- フロントエンド開発者は言う：`string, required` — フォームにあるから
- ドメインの専門家は言う：「実際に誰かが書くタイトル」— 50件の例を見ると、80文字を超えるものはない

最後の答えが*セマンティクス*から来ている。他は習慣から来ている。

## 各ステップの役割

### ストーリー：意図に根ざす

「ユーザーとして、〇〇したい。なぜなら〜」という形式のユーザーストーリーは、あらゆる機能の背後にある*なぜ*を言語化することを強制する。これが土台だ。

「完了状態でフィルタリングしたい、なぜなら残りのタスクだけに集中したいから」というストーリーが教えてくれること：
- 「完了状態」というコンセプトが存在する
- それは二値（完了 / 未完了）だ
- それは記録のためではなく、集中のために存在する

これがすべての後続ステップを形作る。

### ALPS：実装の前に語彙を

ALPS（Application-Level Profile Semantics）は、アプリケーションが*何ができるか*と、使う語彙を——どんな技術とも独立に——記述する。

「アクションは何か？データ要素は何か？どう関連するか？」に答える。

「RESTエンドポイントは何か？」や「データベーステーブルは何か？」には答えない。

ALPSは設計者・開発者・ステークホルダー間の共通言語だ。`doCompleteTodo` が何を意味するかについて全員が合意していれば、実装は翻訳作業になる。

### フェイクデータ：規定の前に観察を

これが最も直感に反するステップで、おそらく最も価値あるステップだ。

制約を定義する前に、50件以上のリアルなフェイクデータを生成する。ランダムなテストデータではなく——実際のユーザーが作るようなリアルなデータを。

そして*眺める*。

タイトルの長さの実際の分布は？メモが null のものはどれくらいあるか？「完了した」todosはどう見えるか？

この観察が問いを変える。抽象的なルールから制約を規定するのをやめ、観察した現実から*発見*し始める。

### 合意：共通理解

フェイクデータのレビューは一人の作業ではない。ステークホルダーとの対話だ。

「この50件のtodosを見てください。正しそうですか？何か足りないものは？何かおかしいものは？」

このステップは、実際のコードが一行も書かれる前に全員を揃える。ここで誤解を発見するコストはゼロに等しい。

### スキーマ：現実を記録する

ここまで来れば、JSONスキーマはほぼ自然に書ける。

`maxLength: 80` は、観察した最長タイトルが41文字だったから（VARCHARのデフォルトだから255、ではなく）。

`"memo": { "type": "string" }` はオプションだ——フェイクデータの多くのtodosにメモがなく、それがリアルだから。

スキーマは規定ではない。合意した現実の*記録*だ。

### Be実装：ドメインを映すコード

スキーマを契約として、Be Frameworkの実装は直感的になる。各`Being`はドメインコンセプトを表す。各`Final`は結果を表す。

コードはデータベースではなく、ドメインのように読める。

## なぜこれが重要なのか

代替手段——いきなり実装に飛びつく——は、技術的には正しいがセマンティクス的には間違ったコードを生む。テストを通過するが、ドメインにはしっくりこない。「コードが何をするか」と「ビジネスが何を意味するか」の間で、常に翻訳が必要になる。

be-semanticは、そのギャップが生まれる前に閉じるための規律だ。

## より深い哲学

良いフレームワークがどう設計されるかに、同じ並行線がある。

最良のフレームワーク——BEAR.Sunday、Be、ALPS自体——は「どのパターンを実装すべきか？」からは生まれなかった。ドメインを観察し、深く理解し、その理解からパターンを浮かび上がらせることで生まれた。

パターンはドメインに従う。逆ではない。

これがbe-semanticの精神だ：制約を現実から浮かび上がらせよ。設計を意味に従わせよ。
